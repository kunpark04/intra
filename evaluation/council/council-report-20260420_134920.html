V4 Leak Isolation — Council Verdict

2026-04-20 13:49 CDT · intra/MNQ trading pipeline · follow-up to 04:30 council on random-K null test
The fork: Phase 1 of the random-K null test blew up on two problems. (1) v11 parquet has no datetime columns — plan's pre-2024 temporal holdout is structurally impossible. (2) Deeper leak discovered: V4 uses global_combo_id as a trained LightGBM categorical, meaning it memorized per-combo win rates. Four options were proposed: (1) combo-agnostic V4, (2) entry-bar temporal holdout, (3) both, (4) accept shipped V4 with caveat.

All five advisors picked Option 1. All five anonymized peer reviewers independently ranked the Contrarian strongest and the Expansionist as the biggest blind spot. Options 2, 3, 4 were unanimously rejected.

Where the Council Agrees

  • Option 1 (combo-agnostic V4) is the only defensible choice. Temporal holdout doesn't fix ID memorization; combined refit doubles compute for no added signal; shipping with a caveat is indefensible.
  • The global_combo_id feature existing in a production filter is the actual headline finding — more important than the null test it was meant to enable.
  • Options 2, 3, 4 all fail on the same grounds: none address the memorization channel.

Where the Council Clashes

Is the shipped Sharpe 2.13 currently trusted?

  • Contrarian Outsider No — ship decision needs re-litigation before any null test.
  • First Principles Expansionist Yes, tacitly — the pipeline shipped; this is learning, not a gate.
  • Executor Neutral — execute steps, null follows.

Chairman sides with Contrarian/Outsider: unanimous peer-review convergence on B-strongest confirms ship-litigation comes first.

Blind Spots the Council Caught

Universal miss (4 of 5 reviewers): audit_full_net_sharpe — the raw-Sharpe top-50 selector — is computed on the same v11 sweep V4 was trained on. The top-50 basket selection and V4's training labels share in-sample trades. Even after a clean combo-agnostic V4 refit, the null test's "shipped-50" anchor is drawn from a leak-contaminated list. Fixing V4 alone is necessary but insufficient.

Single-reviewer catches:

  • V3 (documented as production in CLAUDE.md Phase 5D) may also carry global_combo_id. Leak audit must begin with a grep across all production models, not just V4.
  • No rollback / provisional-status mechanics proposed for tasks/ship_decision.md if the audit invalidates the shipped edge.

The Recommendation

Do Option 1, adopting the Contrarian's re-sequencing and the reviewers' caught blind spots:

  1. Scope the leak — grep V3, V4, and ancillary models for global_combo_id as a categorical/ID feature before retraining.
  2. Mark tasks/ship_decision.md PROVISIONAL pending audit outcome (single-line flag at top).
  3. Refit V4 combo-agnostic: drop global_combo_id from ALL_FEATURES, CATEGORICAL_COLS, and the LightGBM categorical_feature arg. No other changes.
  4. Re-run s6_net on the shipped top-50 with refit V4. Ship-blocker audit.
  5. Only if step 4 CI overlaps the shipped 2.13 do you proceed to the random-K null test.
  6. Plan a second-round null that breaks the audit_full_net_sharpe leak (combo-agnostic selector or held-out combo pool) — otherwise even a passing null under refit V4 only proves half the story.

The One Thing to Do First

Grep the production model training scripts for global_combo_id as a feature, before retraining anything. This scopes the audit. If V3 also has it (CLAUDE.md documents V3, not V4, as production), the rebuild is broader. If only V4, the fix is contained. Ten minutes of grep prevents committing to a retrain scope that's wrong.

Contrarian The framing is wrong, and all four options are bandaids on a deeper rot 5× strongest

You just discovered that V4 — the filter inside your shipped Sharpe 2.13 system — is a combo-ID memorizer. That is not a leak to isolate for the null test. That is a leak in the production pipeline you already declared ship-ready yesterday.

Ask the uncomfortable question: if V4 memorizes global_combo_id, what does the Sharpe 2.13 actually measure? It measures how well LightGBM recalls per-combo base rates on combos it was trained on. The filter isn't filtering — it's retrieving. The 43% ruin on full-pool isn't evidence that concentration is load-bearing; it's evidence that V4 can't rank combos it has over-memorized once you stop pre-filtering by raw Sharpe.

Option 1 isn't a null-test prep step. It's a ship-blocker audit. Retrain combo-agnostic V4, re-run s6_net on the shipped top-50, and see if Sharpe 2.13 survives. If it does, then run the null test. If it collapses, you didn't have a shippable system yesterday.

Do Option 1. But reframe it: not as null-test prep, as ship-decision re-litigation. The null test waits.

First Principles You're asking "which null test" when the real question is "what would change our mind?"

The ablation already told you UCB ranking ≡ random. The full-pool collapse told you top-K concentration is load-bearing. Two established facts. The null test asks: "are the shipped 50 better than 50 random from the same pool?" If NO: concentration is the mechanism, combo identity is noise. If YES: the only remaining mechanism is audit_full_net_sharpe itself having OOS signal — a well-known weak-but-nonzero effect.

You're treating this as a validation gate for shipping. It isn't. You already shipped. This is a mechanism attribution experiment, and mechanism attribution demands you isolate one variable at a time.

Option 2 alone is worthless — you've identified global_combo_id as the memorization channel. Option 3 is 2× compute to answer a question you don't have. Option 4 concedes the leak.

Parquet having no datetime columns is the universe telling you temporal holdout was never the real experiment. Drop ID, retrain, run random-K. One variable. One answer.

Outsider The framing buries the lede — you're deciding whether a null test can even answer your actual question

A model that trained on global_combo_id as a categorical cannot answer "are these 50 better than 50 random." It has memorized which IDs win. Running random-K through it measures "how well did V4 memorize each basket," not "are these combos inherently better." Option 2 is dead on arrival. Option 4 ships a conclusion you can't defend.

Outsider question nobody's asking: why does a production filter have combo ID as a feature at all? That's not a filter — that's a hash table with gradient boosting bolted on. If the ID feature genuinely helps OOS, your pipeline isn't learning generalizable trade quality, it's learning "combo 4471 wins, combo 9823 loses." That should terrify you regardless of the null.

Pick Option 1. If combo-agnostic V4 still produces Sharpe ~2 on the shipped 50, you've proven non-ID structure. If it collapses, you've learned your shipped edge is partly ID-memorization. Treat the ID-feature discovery as the headline, not a footnote.

Executor Option 1. Ship it today.

Poll at 600s. (4) When it finishes, run the 500-basket null test against this refit V4.

Option 2 is actively broken — temporal holdout does NOT fix combo-ID memorization. Option 3 is Option 2's waste plus Option 1's work. Option 4 ships a known-confounded null with a caveat — looks like due diligence while providing none.

One fix, one retrain, one clean null. Don't let "most conservative" masquerade as "most rigorous" when half of it is provably useless.

Expansionist The framing is too small — this is a product pivot hiding in plain sight 5× blind spot

A V4 that only sees parameter-space features is no longer a combo memorizer. It's a combo generator's scoring function. That changes the ceiling dramatically:

(1) Unshackle from the 13,814-combo pool. Sample new combos outside the v11 grid — denser around top-50, or unexplored regions. The sweep becomes your training set, not your universe.

(2) Null test becomes almost-free bycatch. Run it in parallel with a V4-guided active search proposing 500 synthetic combos. If one survives s6_net at Sharpe > 2.13, the pipeline generalizes — the actual scientific claim.

(3) "50 is load-bearing" stops being a black box. A combo-agnostic scorer lets you ask why 50 — parameter coverage? diversification? Push to Sharpe > 2.5 via diverse high-scorers.

Ship Option 1, then scope the active-search extension. 10× outcome hiding here.
